D1：把任务分解成一个抽象 skill graph，但不是纯自由生成，而是模板引导 + 渐进进化

具体逻辑:
先基于主流 IoT benchmark/task ，整理出一版 template skill(对应多种任务种类)
这些 template 不是最终答案，而是高频任务结构的初始模板
D1 分解时，agent 参考模板去生成中间产物
如果任务和模板不完全一致，允许局部修改或补充节点
随着成功中间产物增多，再从中抽象出更一般的 reusable skill
skill 库不断增大，形成持续进化过程

D2：把 D1 生成的 skill graph 映射到当前已有设备空间，对那些超出当前设备空间的模块，生成缺失的实现

情况 1：已有设备包含所有的模块
直接绑定到已有设备/能力

情况 2：已有设备解决不了
说明出现了缺失模块、能力，这时就把这些节点标出来，进入补全生成


skill graph 里先有抽象功能需求
某些节点没法直接映射到已有设备能力空间
就需要生成新的模块代码去补齐
这里可以调用别人的工作


ChatIoT：适合 COTS 、 TAP 、家庭自动化规则类模块
IoTPilot：适合代码生成、修复、补全实现类模块
以后也可以接 repo-wrapping、tool-making 之类的
